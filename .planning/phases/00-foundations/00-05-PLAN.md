  <action>
    Create .planning/field-sessions/recompose-baseline-PENDING/METHODOLOGY.md. Heading: `# Operate-Tab Recomposition Baseline Methodology`. Required sections:

    - "## Why" — one paragraph: PITFALLS Pitfall 6 names the recomposition storm as the primary risk of splitting one StateFlow into five; the Recompose-Count Gate in the promotion checklist enforces a regression threshold equal to this baseline.
    - "## What We Measure" — bullet list:
      - Target: the Operate screen (`app/src/main/java/net/ft8vc/app/ui/operate/OperateScreen.kt`) at root composition.
      - Duration: one full slot cycle (~15 seconds wall clock; one RX slot + the boundary into the next).
      - Metric: total recomposition count of OperateScreen and its top-level children (DecodeListPanel, OperateStatusBar, OperateControls, Waterfall) during the observation window.
      - Conditions: app in foreground; Digirig connected; Operate tab active; at least one decode arriving during the window; no PTT activity (RX-only measurement).
    - "## Lightweight Method (RECOMMENDED for Phase 0)" — numbered steps:
      1. Install the debug APK on the reference device: `./gradlew :app:installDebug`.
      2. Connect device to Android Studio via ADB.
      3. Tools → Layout Inspector → connect to net.ft8vc.debug process.
      4. In Layout Inspector, enable "Highlight Recompositions" (gear icon → Show recomposition counts).
      5. Open Operate tab; tap Start; wait for first decode to arrive.
      6. **Note the baseline composition count for each tracked composable.**
      7. Start a 15-second timer at a fresh slot boundary (UTC :00, :15, :30, or :45 — check device clock).
      8. After 15 seconds elapse, record the final composition count.
      9. **Compute delta = final − baseline.** This is the recomposition count for one slot cycle.
      10. Repeat 3 times; record the median. Median is the committed baseline number.
    - "## Optional Heavier Method (DEFER to Phase 5)" — one paragraph: an instrumented test under `app/src/androidTest/` using `androidx.compose.ui:ui-test-junit4` and a custom RecomposeCounter pattern could automate this, but introducing instrumented-test infrastructure is scope creep for Phase 0 and is intentionally deferred to Phase 5 (which already touches Compose stability). The Phase 5 plan can revisit this if the manual method proves too noisy.
    - "## Output Format" — describe what the captured baseline directory must contain:
      - `README.md` — operator notes (device model, Android version, app version code, Digirig firmware, dial frequency at capture time).
      - `baseline-number.txt` — single integer, the median recomposition count from step 10.
      - `runs/` — three subdirectories `run-1.txt`, `run-2.txt`, `run-3.txt`, each containing the raw (baseline, final, delta) triple from one measurement run.
      - Optional `screenshots/` of the Layout Inspector showing the highlighted recompositions.
    - "## Acceptance Criteria for the Captured Baseline" — bullet list:
      - Three runs completed.
      - Median delta is a non-negative integer ≤ 200 (sanity bound; if the pre-refactor monolith somehow recomposes >200 times per slot, the methodology probably mis-measured and should be re-run).
      - Each run's delta is within ±30% of the median (otherwise: noisy measurement; retry).
      - Captured during RX-only (no TX cycle in the window).
    - "## How Subsequent Phases Use This Baseline" — bullet list:
      - Every refactor phase (1-7) re-runs the same methodology after the phase commits and records its own delta in the PR body.
      - Promotion checklist Recompose-Count Gate fails if the post-phase delta exceeds the Phase 0 baseline by more than ε (suggested ε = 0). Any increase requires written justification.
      - Phase 5 in particular MUST capture a post-refactor recompose count to validate that the `combine + distinctUntilChanged` pattern (PITFALLS Pitfall 6) did not regress the storm.
  </action>
